# 01. 카테고리 체계

- 본 문서는 report-match 파이프라인의 **정산 항목 목록**과 **항목별 필요 서류**를 정의하는 **기준 문서**
- 하위 파이프라인(TASK 02 기업용어매핑룰, TASK 03 JSON 스키마, TASK 07 양식 매핑, TASK 08 누락 증빙 체크)은 본 문서의 **ID**를 참조함
- 이름은 바뀔 수 있으나 **ID는 불변**, 파이프라인의 다른 파일·코드는 ID만 사용
- 작성 근거
  - `data/사후원가정산 샘플(5개)/*/3. 정답지(보고서 초안)/*.xlsx` 5건의 `Ⅵ.내역서` 시트 교차 분석
  - `data/사후원가정산 샘플(5개)/샘플1(...)/2. 업무 기준 정의서(Guidelines) - 공통/비목 분류 지침 및 가이드북/사후원가정산 지침.pdf`
  - `... / 사후원가정산 적격증빙자료 안내가이드.pdf`

## 1. 설계 원칙

- **ID 불변, 이름 가변** — 파이프라인의 모든 외부 참조는 ID로만 수행, 이름 변경은 본 문서 수정만으로 전파
- **parent_id로 계층 표현** — 상위 계층 추가 시 기존 항목의 `parent_id`만 갱신, ID 자체는 변경 금지
- **유형 라벨링 우선, 구체 항목은 확장 대상** — `CAT-A*` 비목(대분류)은 공사 지침 기반으로 고정, 중·소분류(`CAT-A*-###`)는 프로젝트별로 추가 허용
- **상위 분류 불명확 항목은 기타 버킷에 수용** — 매핑 불가 항목은 폐기하지 않고 `CAT-A99` / `DOC-B99`로 회수

- 정답지의 `세부 항목(D)` / `세부 내용(E)`에 해당
- 프로젝트별 편차가 커서 본 문서에서는 **자유 서술 허용**, ID 부여 선택적
- 필요 시 `CAT-A03-001-001` 형식으로 확장 가능

## 4. 변경 시나리오 검증 (TASK 01 검증 기준)

- 본 절은 `지침 변경 → 본 파일 단독 수정 → 파이프라인 반영`이 성립함을 시나리오별로 입증

### 시나리오 1. 하위 항목 추가

- 지침 변경 예: "영상 제작" 직무가 신규 직무로 분리되어 인건비 하위에 추가됨
- 수행 단계
  1. 본 문서 `2.2 중분류` 표에 1행 추가: `CAT-A01-004 | 영상 제작 | CAT-A01 | 샘플3 "영상 제작"`
  2. 다른 파일 수정 **불필요** — `02_기업용어매핑룰.md`는 기업 표현 → 기존·신규 ID 매핑만 갱신, `03_json스키마.json`은 항목 ID 필드가 오픈 enum 형식이므로 수정 불필요
- 입증 방법
  - 본 문서 Diff에 행 추가 1건만 보이면 시나리오 성립
  - TASK 14(매핑 룰 적용률 측정)에서 신규 ID 출현률이 0→양수로 바뀌는지로 확인

### 시나리오 2. 항목 이름 변경

- 지침 변경 예: 공사가 "진행경비" 용어를 "운영경비"로 개칭
- 수행 단계
  1. 본 문서 `CAT-A03`의 **이름 칸만** "경비" → "운영경비"로 수정 (ID 유지)
  2. 매핑 룰(`02_기업용어매핑룰.md`)은 **수정 불필요** — 매핑 키는 ID이지 이름이 아님
- 입증 방법
  - 본 문서 외 변경 없이 리포트의 표기만 갱신됨
  - 파이프라인 테스트로 샘플 5건 재실행 시 이름만 바뀌고 매핑 결과 동일

### 시나리오 3. 상위 계층 추가

- 지침 변경 예: 공사가 "사업비" 최상위 계층을 도입, 기존 4개 비목을 그 하위로 묶음
- 수행 단계
  1. 본 문서 `2.1 대분류` 표에 행 추가: `CAT-A00 | 사업비 | — | 최상위 분류`
  2. 기존 CAT-A01·A02·A03·A04의 `parent_id`를 `—` → `CAT-A00`으로 갱신
  3. ID는 **모두 불변** — 다른 파일은 수정 불필요
- 입증 방법
  - 본 문서 Diff에 행 추가 1건 + parent_id 4칸 수정만 보이면 성립
  - 하위 파이프라인(JSON 스키마·양식 매핑)의 ID 참조가 그대로 유효

## 5. 확장·편집 가이드 (설명서 연계)

- `기준파일/설명서.md`의 "새 항목이 생겼어요" / "항목 이름만 바뀌었어요" / "새 증빙 서류가 필요해졌어요" 절과 대응
- 편집 시 원칙
  - ID 재할당 **금지** (삭제된 ID도 재사용 금지, 필요 시 "폐기" 상태로 표기)
  - 표 행 추가·수정만으로 작업 종결되도록 다른 파일 수정 유발 여부를 항상 체크
  - 수정 후 `data/` 샘플 5건 기준 파이프라인 회귀 테스트 권장
