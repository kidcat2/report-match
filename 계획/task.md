- Task 간 직접 코드 의존 금지, 연결은 오직 산출물(파일/스키마)을 매개로만
- JSON 스키마(Phase A 확정 전)의 임의 확장 금지

## 파이프라인 개관

- 각 단계에서 "누가 무엇을 넣고 무엇이 나오는지"의 공통 참조점
- 개별 Task는 본 개관의 한 칸을 구체화하는 관계, 입출력 반복 서술 불필요
- "사용자" = 홈페이지에 파일을 업로드하는 주체 (기업 담당자 또는 당사 실무자)
- "개발자" = 파이프라인 구축·운영자, 참조 파일을 시스템에 내재화
- 사전 주입 파일의 실체는 `기준파일/` 폴더에 보관 (계획 폴더 밖, 사용자 편집 허용 공간)
- `기준파일/설명서.md`에서 각 파일의 역할·수정 가이드 확인 가능

| 단계                    | 사용자가 넣는 것              | 개발자가 사전 주입                                  | 시스템 산출물              |
|-------------------------|-------------------------------|------------------------------------------------------|----------------------------|
| Phase 1 (웹 → LLM)      | 기업 제출 이미지 서류         | `기준파일/03_json스키마.json`, LLM 프롬프트         | `<company>_raw.json`       |
| Phase 2 (JSON → 중간 MD)| —                             | `기준파일/01_카테고리체계.md`, `기준파일/04_중간산출물_양식.md`, 매핑 프롬프트 | `<company>_output.md`      |
| Phase 3 (MD → XLSX)     | —                             | `기준파일/05_엑셀매핑규칙.md`, `기준파일/06_엑셀양식.xlsx`, 양식 작성 프롬프트 | `<company>_final.xlsx`     |
| 병렬: 누락 증빙 체크     | —                             | `기준파일/01_카테고리체계.md` (필요 증빙 섹션)      | `<company>_missing.md`     |

- Phase 1 출력은 Phase 2 입력으로 그대로 연결 (사용자 재업로드 불필요)
- Phase 2 출력은 Phase 3 입력으로 그대로 연결
- 누락 증빙 체크는 Phase 1 또는 Phase 2 산출물 어느 쪽으로도 수행 가능, Phase 3와 독립 병렬 동작

## Phase A. 분석·설계 (선행)

### TASK 01 - 카테고리 체계 조사 및 양식 설계

- **목적**: 업무기준정의서·정답지로부터 정산 항목 체계를 추출, 기업별 중간 정리본 템플릿이 될 영속적 MD 양식을 설계
- **Input**: 업무기준정의서 1건 (샘플 간 동일하므로 대표 1건), 정답지 5건 전체
- **Output**: `기준파일/01_카테고리체계.md` — **프로젝트 필수 포함 파일**, 기업별 중간 산출물(`<기업명>_output.md`)의 양식 기준
- **Steps**: (1) 업무기준정의서 1건 분석 → (2) 정답지 5건 교차 분석 → (3) 섹션·항목 계층 식별 (예: 금액 정산 항목, 항목별 필요 서류 등) → (4) 이름 변경·하위 추가·상위 계층 추가에 견디는 구조로 설계 (항목 ID와 이름 분리) → (5) MD 작성 후 5개 샘플로 검증
- **검증 방법**: Given 지침 변경 시나리오 3종 (하위 항목 추가 / 항목 이름 변경 / 상위 계층 추가) / When `기준파일/01_카테고리체계.md` 단일 파일만 수정 / Then 다른 Task의 코드·문서 수정 없이 변경이 하위 파이프라인에 반영 가능함을 문서로 입증
- **Assumptions**: `[ASSUMPTION: 최종 섹션 구성은 조사 중 확정, 초안 단계에서는 "금액 정산 항목"과 "항목별 필요 서류" 2개 섹션을 가정]`, `[ASSUMPTION: 이름 변경에도 매핑 유지를 위해 항목별 고정 ID 부여]`

### TASK 02 - 기업 표현 → 카테고리 매핑 룰 설계

- **목적**: 기업마다 다른 항목명·표현을 당사 카테고리체계 항목 ID로 번역하는 규칙 설계
- **Input**: `기준파일/01_카테고리체계.md`, `data/사후원가정산 샘플(5개)/` (기업별 제출 자료)
- **Output**: `기준파일/02_기업용어매핑룰.md` (기업 표현 → 카테고리 항목 ID 매핑 테이블 + 우선순위·예외 규칙)
- **Steps**: (1) 샘플 5개에서 기업 측 항목명·표현 수집 → (2) 1:1 직접 매핑 정리 → (3) 다대일·애매 케이스의 우선순위 룰 정의 → (4) 어느 카테고리에도 속하지 않는 예외 규칙 정의 → (5) 샘플 5개로 룰 적용 검증
- **검증 방법**: Given 샘플 5개 / When 룰 적용 / Then 기업 측 모든 항목이 카테고리 항목 ID 1건 이상과 매칭되거나 예외 목록에 기록됨
- **Assumptions**: `[ASSUMPTION: 샘플 5건 기준 룰 수립, 신규 기업 추가 시 룰 확장 필요]`, `[ASSUMPTION: 매핑 키는 카테고리 항목 ID(이름 아님)로 고정하여 이름 변경에 영향받지 않음]`

### TASK 03 - JSON 스키마 정의

- **목적**: 로컬 LLM이 출력할 JSON 구조 확정 (B·C·D 공통 계약)
- **Input**: `기준파일/02_기업용어매핑룰.md`, `data/사후원가정산 샘플(5개)/`
- **Output**: `기준파일/03_json스키마.json` + 예시 샘플 JSON 1건
- **Steps**: (1) 매핑 룰 기반 필드 목록화 → (2) 계층·타입·필수 여부 정의 → (3) 예시 JSON 작성 → (4) 스키마 검증
- **검증 방법**: Given 스키마 파일 / When 예시 JSON을 JSON Schema 검증 / Then 검증 통과

## Phase B. Jetson 인프라

### TASK 04 - FastAPI 뼈대 + 업로드·다운로드 플로우 (PC 로컬)

- **목적**: PC 환경에서 업로드 → stub LLM → 결과 다운로드의 최소 웹 왕복 구현
- **Output**: PC 로컬 구동 FastAPI 서버 + 단일 업로드 UI 페이지
- **Steps**: (1) FastAPI 프로젝트 초기화 → (2) 단일 업로드 칸 UI 페이지 (기업 파일만) → (3) 업로드 엔드포인트: 파일 수신 → 임시 저장 → stub JSON 응답 저장 → (4) 다운로드 엔드포인트: 결과 JSON 반환 → (5) PC 로컬 브라우저(`http://127.0.0.1:<port>`)에서 업로드·다운로드 왕복 확인
- **검증 방법**: Given PC 로컬에서 기동된 서버 / When 브라우저에서 기업 파일 1건 업로드 → 다운로드 클릭 / Then stub 응답 JSON이 다운로드됨
- **Assumptions**: `[ASSUMPTION: FastAPI 프레임워크]`, `[ASSUMPTION: 업로드 칸은 1개 — 기업 파일만, 기준파일·스키마는 서버 내부에서 로드]`, `[ASSUMPTION: MVP는 단일 파일 처리, 다중 업로드·인증·파일 정리는 task_확장.md]`

### TASK 05 - 상태 조회 + 기준파일 자동 주입

- **목적**: 처리 상태 가시화와 서버가 `기준파일/`을 자동 로드해 LLM 프롬프트에 주입하는 메커니즘 구축
- **Input**: TASK 04 완성 서버
- **Output**: 상태 조회 엔드포인트 + 기준파일 자동 로드·프롬프트 주입 로직이 결합된 서버
- **Steps**: (1) 처리 상태 5단계 정의 (대기중·전처리중·추론중·완료·실패) → (2) 상태 조회 엔드포인트 (요청 ID 기준) 구현 → (3) UI에서 상태 폴링·표시 → (4) 서버 시작 시 `기준파일/01_카테고리체계.md`·`기준파일/03_json스키마.json` 로드 → (5) stub LLM 호출부에 로드한 기준파일 내용이 프롬프트로 전달되는지 로그로 검증
- **검증 방법**: Given 기동된 서버 / When 파일 업로드 → 상태 폴링 / Then 상태가 대기중 → 전처리중 → 추론중 → 완료 순으로 전환됨 + 로그에 기준파일 내용이 프롬프트로 포함되었음 확인
- **Assumptions**: `[ASSUMPTION: 요청 ID는 UUID4]`, `[ASSUMPTION: 상태 저장은 인메모리 dict, 재시작 시 휘발 — 영속화는 task_확장.md]`

### TASK 06 - Jetson 배포 및 end-to-end 검증

- **목적**: PC에서 개발된 서버를 Jetson Orin에 git 배포하고 로컬 브라우저 기준 end-to-end 동작 검증
- **Input**: TASK 04, TASK 05 완성 서버
- **Output**: Jetson 구동 웹서버 + 배포 절차 문서 `계획/산출물/B06_webserver_배포.md`
- **Steps**: (1) `requirements.txt` 정의 및 PC/Jetson 공통 의존성 확인 → (2) PC에서 `git push` → (3) Jetson에서 `git pull` + `pip install -r requirements.txt` → (4) `.env`·비밀키 Jetson 수동 배치 → (5) Jetson에서 서버 기동 → (6) Jetson 로컬 브라우저에서 업로드 → 상태 → 다운로드 왕복 확인
- **검증 방법**: Given Jetson에 배포된 서버 / When Jetson 로컬 브라우저에서 기업 파일 1건 업로드 / Then 상태가 완료로 전환되고 stub JSON 다운로드 가능
- **Assumptions**: `[ASSUMPTION: 개발은 PC(Windows), 실행은 Jetson(Ubuntu, ARM64)]`, `[ASSUMPTION: MVP 범위 — Jetson 로컬 접속만, 외부 공개·HTTPS·인증·부팅 자동 실행은 task_확장.md]`

### TASK 07 - 이미지 전처리 파이프라인

- **목적**: 샘플 폴더 내 다양한 확장자 원본 파일을 LLM 입력에 적합한 PNG로 일괄 변환·정규화
- **Input**: `data/사후원가정산 샘플(5개)/<샘플명>/증빙자료 원본/`, `data/사후원가정산 샘플(5개)/<샘플명>/산출내역서/`
- **Output**: `./preprocessed/<샘플명>/*.png` (포맷 통일 + deskew·크롭·해상도 정규화)
- **Steps**: (1) 확장자별 변환 규칙 정의 (PDF→`pdf2image`/PyMuPDF로 페이지별 PNG, JPG·PNG·기타 이미지→PNG 재저장) → (2) 전처리 항목 정의 (deskew / 여백 제거 / DPI 정규화) → (3) 함수 API 설계 `preprocess(input_dir, output_dir)` (경로 파라미터화, 샘플·웹업로드 양쪽 재사용) → (4) 샘플 5개로 테스트
- **검증 방법**: Given 혼합 확장자 샘플 폴더 / When `preprocess` 실행 / Then 모든 파일이 PNG로 변환되어 `./preprocessed/<샘플명>/` 아래 저장됨
- **Assumptions**: `[ASSUMPTION: 대상 확장자는 PDF, JPG, PNG, HEIC 등 일반 문서 이미지 포맷]`, `[ASSUMPTION: 엑셀 파일은 본 Task 범위 제외, 필요 시 별도 파서 분기]`

### TASK 08 - 웹서버 ↔ 실제 LLM 통합

- **목적**: TASK 04~06에서 쓰던 stub을 실제 로컬 LLM 호출로 교체, 전처리와 연결하여 end-to-end 완성
- **Input**: TASK 03 스키마, TASK 06 완성 서버, TASK 07 전처리, Phase D 선정 모델
- **Output**: 실제 LLM이 연결된 end-to-end 파이프라인
- **Steps**: (1) 업로드 이벤트 → TASK 07 전처리 호출 → (2) 전처리 결과 → 실제 LLM 호출 → (3) LLM 출력 저장 → 다운로드 링크 노출 → (4) 단건 처리 시간 측정
- **검증 방법**: Given Jetson 배포 서버 + 선정 모델 / When 기업 파일 1건 업로드 후 실행 / Then 5분 이내 실제 LLM이 생성한 JSON 다운로드 가능
- **Assumptions**: `[ASSUMPTION: 단건 처리 상한 5분]`

- **Steps**: (1) 지정 폴더 JSON 로드 → (2) 매핑 룰을 Claude 프롬프트에 주입 → (3) Claude API 호출로 필드 매핑 → (4) 양식 파일 생성
- **검증 방법**: Given 샘플 JSON 1건 / When 실행 / Then 양식 파일 생성 + 주요 필드 채움 확인 (체크리스트)
- **Assumptions**: `[ASSUMPTION: 출력 포맷 xlsx]`

### TASK 10 - 누락 증빙 체크

- **검증 방법**: Given 샘플 1건 / When 실행 / Then 누락 + 제출 항목의 합집합이 필요 항목 전체와 일치
- **Assumptions**: `[ASSUMPTION: 제출 파일의 파일명 규칙 또는 매핑 메타 존재]`

## Phase D. 로컬 LLM 모델 실험

### TASK 11 - 모델 후보 환경 구축

- **목적**: Jetson Orin에 후보 로컬 LLM 각각 구동 환경 확보
- **Input**: Jetson 하드웨어 사양, 후보 모델 리스트
- **Output**: `계획/산출물/D11_모델환경.md` (설치 절차·버전·VRAM 기록)
- **Steps**: (1) 후보 모델 확정 → (2) 각 모델 설치 및 최소 추론 확인 → (3) 환경 격리 (컨테이너 or venv)
- **검증 방법**: Given 각 후보 모델 / When 단일 이미지 추론 / Then 정상 출력 + VRAM 측정치 기록
- **Assumptions**: `[ASSUMPTION: 후보는 Qwen-VL, Gemma-Vision 등 비전 지원 모델]`

### TASK 12 - 모델별 프롬프트 튜닝

- **목적**: JSON 스키마(TASK 03)에 따라 안정 출력되도록 모델별 프롬프트 최적화
- **Input**: TASK 03 스키마, TASK 11 환경
- **Output**: 모델별 프롬프트 `./prompts/<model>.txt`
- **Steps**: (1) 초기 프롬프트 작성 → (2) 샘플 5개로 반복 튜닝 → (3) 스키마 검증 통과율 측정
- **검증 방법**: Given 튜닝된 프롬프트 / When 샘플 5개 실행 / Then 5건 모두 JSON 스키마 검증 통과

### TASK 13 - 모델 비교 실험

- **목적**: 동일 샘플에 대해 모델별 정확도·속도·메모리 비교

- **Output**: `계획/산출물/D13_모델비교.md` (지표 표 + 선정 근거)
- **Steps**: (1) 측정 스크립트 작성 → (2) 각 모델 3회 반복 실행 → (3) 평균 지표 집계 → (4) 선정 근거 기록
- **검증 방법**: Given 2종 이상 모델 / When 측정 스크립트 실행 / Then 모델별 정확도·처리시간·VRAM 표 생성

### TASK 14 - 선정 모델 Jetson 최종 배포

- **목적**: 선정 모델을 Jetson Orin에 안정 배포 (메모리 관리·크래시 대응 포함)

- **검증 방법**: Given 배포된 모델 / When 연속 30회 추론 / Then 메모리 누수 없음 + 모든 요청 정상 응답
