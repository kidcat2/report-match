# Task 작성 지침

- 본 프로젝트(report-match)의 Task 문서 작성 원칙
- Task 문서 = 실행 수준 문서, PRD의 "무엇을"을 "어떻게 / 어떤 형식으로"로 구체화
- 모든 Task는 단일 파일(`계획/task.md`)에 Phase별 그룹화하여 정리
- PRD 수준의 계획·배경·Phase 간 관계는 본 문서에서 제외

## 1. Task 문서의 목적

- 개별 작업 수행에 필요한 실행 정보 집약
- 한 Task = 한 세션(약 30~60분) 내 완료 가능한 크기
- 완료 여부 판단 가능한 **검증 기준** 필수
- 역할 구분 — PRD: "무엇을 / 어떤 단위로 / 어떤 순서로" · Task: "어떻게 / 어떤 형식으로 / 어떻게 검증"

## 2. 필수 구성 항목

### 2.1 최상단 — 공통 제약 조건

- 모든 Task에 공통 적용되는 제약을 문서 최상단에 하나의 블록으로 배치
- 예: 기존 기능 보호, 공용 모듈·스키마 변경 금지, 실행 환경 가정 등

### 2.2 Phase 단위 그룹화

- 각 Task는 `## Phase 1.`, `## Phase 2.` 등 Phase 헤더로 묶음
- Phase 아래 해당 Task들을 순서대로 나열

### 2.3 Task 항목 구조

- **제목** — `TASK 01 - Local LLM 가능 항목 정리` 형식 고정
- **목적** — 1줄 요약 (1줄 초과 금지)
- **입력(Input)** — 참고 파일·데이터·선행 산출물 경로
- **출력(Output)** — 산출물 이름·형식·저장 위치
- **수행 단계(Steps)** — 순서 있는 번호 리스트
- **검증(테스트) 방법** — Given / When / Then 또는 체크리스트
- **가정(Assumptions)** — `[ASSUMPTION: ...]` 태그 (초안 단계 한정)

### 2.4 가정(Assumptions) 운용 원칙

- 초안 단계 — 가정 허용, 불확실성을 태그로 명시
- 본격 진행 단계 — 모든 가정이 해소될 때까지 계획서 보완 반복
- 가정이 남아 있으면 실행 진입 불가

## 3. 작성 원칙

- **Task 독립성** — 각 Task는 다른 Task에 영향을 주지 않도록 설계 (객체지향 원칙). 산출물을 통한 의존은 허용, 단 B Task 수정이 A·C Task 수정을 강제하면 분할 실패
- **목적 1줄 제한** — 목적은 반드시 1줄
- **5~6줄 내 서술** — Task 설명 전체 5~6줄 이내 유지, 초과 시 분할 또는 불필요 내용 삭제
- **확장성 항상 고려** — 기능·모델·데이터 증가 가능성을 전제로 설계
- **수치·예시로 구체화** — "빠르게" 대신 "3초 이내", "정확도 90% 이상" 등 측정 가능 표현
- **하지 말아야 할 것의 분리** — 하지 말아야 할 것 중 제약 조건 성격은 2.1 공통 제약 조건으로 이동

## 4. AI 에이전트 친화적 작성

- **개조식 제목 고정** — `TASK 01 - XXX` 형식 일관 유지
- **구조화된 마크다운** — 헤더 계층 일관, 체크리스트·코드블록 적극 활용
- **상대 경로 원칙** — 프로젝트 루트 기준 상대 경로 사용, 경로 하드코딩 최소화
- **검증 명령 병기** — 실행 명령·테스트 스크립트·수동 체크 절차를 검증 방법 옆에 명시
- **기존 기능 보호** — 명시적 보호 조항 유지, 단 사용자가 기존 기능 변경을 명시적으로 요청한 경우 적용 제외

## 5. 흔한 실수 피하기

- **Task가 너무 큼** — 30~60분 초과면 Phase 수준, 재분할
- **모호한 검증 기준** — "잘 동작한다" 금지, 측정 가능한 조건으로 치환
- **경로 하드코딩** — 경로는 언제든 변경 가능, 상대 경로·변수·설정값 활용, 변경 시 영향 범위를 예측 가능하게 표기
- **검증 방법 누락** — 완료 정의가 문서 내에서 판별 불가능하면 Task 미완
- **숨겨진 가정** — `[ASSUMPTION: ...]`로 명시, 본 진행 단계에 남으면 안 됨
- **PRD 수준 정보 혼입** — 배경·Phase 간 관계·전체 전략은 PRD 몫
