# PRD 작성 지침

- 본 프로젝트(report-match)의 Product Requirement Document 작성 원칙
- PRD = **계획서 수준** 문서, 각 Phase·Task의 세부 작업 방법·형식·절차는 별도 task 문서에서 관리

## 1. PRD의 목적

- 프로젝트의 **목적**과 **해결하려는 문제**를 하나의 문서로 확정
- 세부 작업 방법·포맷·절차는 task 문서 몫, PRD에 수록 금지
- **살아있는 문서** — 방향성 변경 시 PRD부터 갱신, 변경 이력 보존

## 2. 필수 구성 항목

- **개요 / 배경** — 해결하려는 문제와 프로젝트의 목적
- **범위(In Scope)와 비범위(Out of Scope)** — 이번에 만드는 것과 만들지 않는 것의 경계 (비범위 명시가 더 중요한 경우 다수)
- **기능 요구사항** — Phase 레벨의 "이 프로젝트가 수행하는 것"만, 세부 동작·UI·포맷 제외
- **테스트 / 검증 기준** — Phase 종료 여부를 판단하는 조건, 세부 테스트 케이스는 task 문서로
- **Phase별 실행 계획** — Phase A~E의 큰 그림 배치와 의존 관계

## 3. 작성 원칙 (계획서 수준)

- **큰 그림 우선** — "무엇을 / 어떤 단위로 / 어떤 순서로"까지만, "어떻게·어떤 형식으로"는 task 문서로 이관
- **범위 경계선 선명화** — 특히 Out of Scope를 강하게 명시, 경계가 흐리면 이후 task 작성 전체가 불안정
- **Phase 간 의존 관계 명시** — 어느 Phase 산출물이 어느 Phase의 전제인지 한눈에 드러나게
- **용어 통일** — "근거자료", "증빙서류", "정답지" 등 공용 용어는 PRD 앞부분에서 정의, 이후 모든 task 문서가 동일 용어 사용

## 4. AI 에이전트 친화적 작성 (계획서 관점)

- **구조화된 마크다운** — 헤더 계층 일관, Phase·항목 단위 즉시 참조 가능
- **Phase는 독립 섹션으로 분리** — AI 에이전트가 Phase 단위 컨텍스트를 끊어서 읽기 용이
- **외부 참조는 경로로 연결** — data 폴더·task 문서 위치를 상대 경로로 명시, 본문 비대화 방지
- **분량 관리** — 세부 내용이 유입되면 task 문서로 밀어내어 PRD는 개요 수준으로 유지

## 5. 흔한 실수 피하기

- **task 수준 세부가 PRD로 유입** — "이 기능은 이런 포맷으로…"가 나오면 즉시 task 문서로 이동
- **범위 모호** — "나중에 할 수도" 금지, In Scope / Out of Scope 중 택일하여 확정
- **Phase 종료 조건 누락** — 종료 기준 없이는 Phase가 무한 연장, 간단해도 반드시 명시
- **문서 업데이트 소홀** — 실제 진행과 PRD가 괴리되면 계획서 효력 소멸, 방향성 변경 시 PRD 우선 갱신
- **세부 task 구조와의 연결 끊김** — PRD는 task 문서 목록·경로와 연동 필수, 끊기면 계획·실행 분리
